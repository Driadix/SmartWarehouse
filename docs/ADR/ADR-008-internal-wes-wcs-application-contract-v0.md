# ADR-008: Внутренний прикладной контракт `WES <-> WCS` v0

**Статус:** Proposed  
**Дата:** 2026-04-04  
**Связанные артефакты:** ArchitecturalVision.md, DomainModel-v0.md, Event-Catalog-v0.md, ADR-003, ADR-005, ADR-006, ADR-007

---

## 1. Контекст

Для `Phase 1` уже зафиксированы:

- внешний бизнес-контракт `Northbound API v0`;
- нижний канонический контракт `Southbound Contract v0`;
- модель полномочий `WES / WCS / ACL`;
- правило, что `WES` планирует макрошаг `ExecutionTask`, а `WCS` материализует его во внутренние runtime-фазы.

При этом до настоящего момента отсутствовал явный внутренний прикладной контракт между `WES` и `WCS`:

- какими командами `WES` передаёт плановый шаг в исполнение;
- какие именно факты `WCS` обязан вернуть в `WES` для продолжения жизненного цикла `Job`;
- как оформляются приостановка, эскалация на перепланирование и корреляция сообщений;
- должен ли этот контракт сразу фиксироваться как отдельный сетевой API.

Без такого решения есть риск:

- неявного обмена через общие таблицы БД;
- дублирования семантики между каноническими событиями платформы и ad-hoc сообщениями реализации;
- привязки логической границы `WES / WCS` к конкретной физической схеме развёртывания.

---

## 2. Решение

### 2.1. Контракт является прикладным и транспортно-независимым

Внутренний контракт `WES <-> WCS` фиксируется как **прикладной контракт уровня платформы**, а не как обязательный сетевой API.

Следствия:

- логическая граница существует всегда, даже если `WES` и `WCS` развёрнуты внутри одного процесса `platform-core`;
- в `Phase 1` рекомендуемой физической привязкой является in-process command/event bus;
- при дальнейшем выносе `WCS` в отдельный процесс тот же контракт может быть привязан к брокеру сообщений без изменения бизнес-смысла.

### 2.2. Команды `WES -> WCS`

Для `v0` фиксируется минимальный набор прикладных команд:

```text
SubmitExecutionTask
CancelExecutionTask
```

#### `SubmitExecutionTask`

Команда публикуется `WES`, когда плановый `ExecutionTask` готов к передаче в контур исполнения.

Минимальный состав полей:

```text
SubmitExecutionTask {
  messageId
  contractVersion
  executionTaskId
  taskRevision
  jobId
  correlationId
  taskType
  assigneeId
  participantRefs[]
  sourceNode?
  targetNode?
  transferMode?
}
```

Правила:

- `executionTaskId` является стабильным идентификатором макрошагa;
- `taskRevision` увеличивается при перепланировании того же шага или при его безопасной замене;
- `WCS` не вправе изменять содержимое плана, переданного в `SubmitExecutionTask`;
- повтор той же команды с тем же `executionTaskId + taskRevision` обязан быть идемпотентным.

#### `CancelExecutionTask`

Команда публикуется `WES`, если жизненный цикл `Job` требует остановки текущего шага исполнения.

Минимальный состав полей:

```text
CancelExecutionTask {
  messageId
  contractVersion
  executionTaskId
  taskRevision
  jobId
  correlationId
  reasonCode?
}
```

`CancelExecutionTask` не означает, что `WCS` обязан мгновенно оборвать уже идущее физическое действие. `WCS` обязан привести шаг к безопасному техническому исходу и только после этого публиковать итоговый переход состояния.

### 2.3. Ответные факты `WCS -> WES`

Для `v0` **не вводится отдельный набор новых событий**, дублирующих уже существующие канонические платформенные факты.

`WES` обязан потреблять следующие канонические события `WCS`:

- `ExecutionTaskStateChanged` — основной факт жизненного цикла шага;
- `TransferCommitted` / `TransferAborted` — при необходимости аудита и диагностики;
- `PayloadCustodyChanged` — для контроля факта передачи груза;
- `DeviceSessionLost` — как технический контекст причины приостановки;
- `FaultRaised` / `FaultCleared` — как технический контекст деградации ресурса.

Ключевой обязательный факт для координации `WES <-> WCS`:

```text
ExecutionTaskStateChanged {
  executionTaskId
  jobId
  previousState
  newState
  reasonCode?
  resolutionHint?
  replanRequired?
}
```

### 2.4. Семантика состояний шага для `WES`

`WES` интерпретирует `ExecutionTaskStateChanged` следующим образом:

- `newState = InProgress` — `WCS` подтвердил фактический старт исполнения;
- `newState = Completed` — постусловия шага подтверждены, `WES` вправе переходить к следующему макрошагу;
- `newState = Suspended` — продолжение невозможно без ожидания, повторного согласования, перепланирования или операторского решения;
- `newState = Failed` — текущий шаг завершён неуспешно и не может быть продолжен как есть;
- `newState = Cancelled` — шаг технически остановлен и закрыт.

Для `newState = Suspended` и `newState = Failed` `WCS` обязан указывать:

- `reasonCode` — машиночитаемую причину;
- `resolutionHint` — одну из подсказок `WAIT_AND_RETRY | REPLAN_REQUIRED | OPERATOR_ATTENTION`;
- `replanRequired` — булево поле, достаточное для простых решений `WES`.

### 2.5. Корреляция и причинность

Для `v0` фиксируются следующие правила:

- `correlationId` планового шага задаётся `WES` и сохраняется `WCS` на всём жизненном цикле `ExecutionTask`;
- `SubmitExecutionTask.correlationId = ExecutionTask.correlationId`;
- все события `WCS`, относящиеся к данному шагу, публикуются с тем же `correlationId`;
- `causationId` указывает на `messageId` прикладной команды или `eventId` канонического факта, непосредственно вызвавшего новый переход;
- причинная упорядоченность требуется в пределах одного `executionTaskId`.

### 2.6. Граница полномочий

Внутренний контракт закрепляет следующие запреты:

- `WES` не публикует runtime-фазы `BoardCarrier`, `MoveCarrier`, `ExitCarrier`, `GrantMotionWindow`;
- `WCS` не меняет `taskType`, `assigneeId`, `transferMode` и целевую точку планового шага без нового `SubmitExecutionTask`;
- прямое чтение `WES` из внутренних runtime-таблиц `WCS` и прямое чтение `WCS` из таблиц планировщика не считается контрактом интеграции.

## 3. Последствия

### 3.1. Что упрощается

- появляется явный прикладной контракт без обязательного преждевременного process split;
- `WES` и `WCS` могут развиваться как отдельные модули внутри одного `platform-core`;
- жизненный цикл `Job` опирается на канонические факты `WCS`, а не на общие mutable-таблицы;
- дальнейший вынос `WCS` в отдельный процесс не требует переосмысления бизнес-семантики.

### 3.2. Что становится обязательным

- `ExecutionTaskStateChanged` должен нести причину и подсказку разрешения для `Suspended` и `Failed`;
- команды `SubmitExecutionTask` и `CancelExecutionTask` должны быть идемпотентны;
- прикладной контракт должен тестироваться независимо от конкретного транспорта.

## 4. Рассмотренные альтернативы

### Альтернатива A: Сразу фиксировать сетевой `WES <-> WCS` API в AsyncAPI/gRPC

Плюсы:

- ранняя машиночитаемая спецификация;
- проще заранее готовить process split.

Минусы:

- преждевременно навязывает физическую схему развёртывания;
- увеличивает объём интеграционного кода ещё до появления реальной необходимости;
- создаёт второй “официальный wire-level API” внутри одной развёртки.

Не выбрана для `Phase 1`.

### Альтернатива B: Координировать `WES` и `WCS` через общую БД без явных команд и событий

Плюсы:

- минимальный стартовый объём кода.

Минусы:

- размывает логическую границу полномочий;
- плохо тестируется;
- плохо мигрирует к отдельным процессам;
- провоцирует скрытую связанность по схеме БД.

Отклонена.

### Альтернатива C: Ввести минимальный прикладной контракт и опереться на уже существующие канонические события `WCS`

Плюсы:

- не дублирует `Event Catalog`;
- сохраняет логическую границу без раннего технологического lock-in;
- хорошо соответствует модульному монолиту `Phase 1`.

Минусы:

- машиночитаемую форму придётся добавить отдельно, если появится обязательный process split.

Выбрана как рекомендуемая для `Phase 1`.
