# ADR-002: Текущий базовый состав реализации, семейства ресурсов и активированные режимы передачи

**Статус:** Принято  
**Дата:** 2026-04-03  
**Связанные артефакты:** Glossary, ArchitecturalVision.md, Architecture-Baseline-Phase-1.md, ADR-001, ADR-003, ADR-004, DomainModel-v0

---

## 1. Контекст

Команда должна начать разработку без реального набора конечных устройств, но с понятной целевой конфигурацией первой поставки. Основной риск состоит в том, чтобы не смешать текущий базовый состав реализации с постоянными архитектурными инвариантами платформы.

Для старта нужен документ, который:

- охватывает первый поставляемый склад;
- проверяет ключевые механизмы платформы;
- не навязывает будущие семейства ресурсов как обязательные уже сейчас;
- не маскирует аппаратную возможность устройства под автоматически активированный режим платформы.

---

## 2. Решение

### 2.1. Семейства ресурсов текущего базового состава

В текущий базовый состав включены следующие семейства и конкретные реализации:

- **`MobileTransportUnit` -> `Shuttle3D`**
- **`VerticalCarrier` -> `HybridLift`**
- **`StationBoundary` -> `LoadStation`, `UnloadStation`**

### 2.2. Семейства ресурсов, не входящие в текущий состав

В текущий базовый состав не входят:

- `Shuttle1D`;
- `DirectStorageCrane`;
- многоместные вертикальные перевозчики;
- сложные сети станций с собственной внутренней логикой оркестрации.

### 2.3. Топологические допущения текущего базового состава

Текущий базовый состав обязан поддерживать:

- уровневые графы с ветвлениями и поворотами;
- `SwitchNode` как узел, требующий исключительного краткосрочного резервирования;
- `TransferPoint` рядом с `CarrierNode`;
- `StationNode` для зон загрузки и выгрузки.

Таким образом, текущий базовый состав не ограничивается линейным рельсовым сценарием и сразу учитывает требования `Shuttle3D`.

### 2.4. Активированные режимы передачи

В текущем базовом составе активированы только следующие режимы передачи:

- `LoadStation -> Shuttle3D`
- `Shuttle3D -> HybridLift -> Shuttle3D`
- `Shuttle3D -> UnloadStation`

Межуровневый режим передачи фиксируется как:

`SHUTTLE_RIDES_HYBRID_LIFT_WITH_PAYLOAD`

### 2.5. Аппаратная возможность и активированный режим передачи

Для текущего базового состава принимается следующее правило:

- аппаратная возможность устройства и активированный режим передачи платформы — не одно и то же;
- `HybridLift` может физически поддерживать передачу грузовой единицы без въезда шаттла, но это не означает автоматической поддержки такого режима платформой;
- включение нового режима передачи требует отдельного обновления базового состава реализации и, при необходимости, нового ADR.

### 2.6. Режимы передачи, не входящие в текущий состав

В текущий базовый состав не включены:

- режим передачи грузовой единицы без въезда шаттла для `VerticalCarrier`;
- режим перевозки нескольких шаттлов одним перевозчиком;
- прямая передача грузовой единицы между перевозчиком и станцией без участия базового шаблона подвижной транспортной единицы.

### 2.7. Правило расширения архитектуры

Добавление нового семейства ресурса или нового режима передачи допустимо только при выполнении трёх условий:

1. сохраняется модель полномочий `WES / WCS / ACL`;
2. появляются формализованные правила исполнения семейства и, при необходимости, новый контроллер семейства ресурсов;
3. базовый состав реализации обновляется явно, а не скрыто через допущения в коде.

---

## 3. Последствия

### 3.1. Что становится возможным уже сейчас

Текущий базовый состав достаточен, чтобы начать разработку:

- `Topology Compiler`;
- построение маршрутов по ветвящемуся графу;
- `WCS Execution Core`;
- контроллеров семейств ресурсов для `Shuttle3D` и `VerticalCarrier`;
- имитационной модели нижнего контура;
- `Digital Twin`;
- сквозных сценариев от `LoadStation` до `UnloadStation`.

### 3.2. Что сознательно откладывается

Не требуется сейчас:

- проектировать семантику хранения `DirectStorageCrane`;
- поддерживать режим передачи грузовой единицы без въезда шаттла для `VerticalCarrier`;
- вводить универсальные абстракции для всех возможных будущих семейств ресурсов.

### 3.3. Что должно оставаться расширяемым

В то же время текущий базовый состав не должен зашивать следующие жёсткие инварианты:

- что любой вертикальный транспорт всегда является лифтом;
- что грузовая единица всегда принадлежит подвижной транспортной единице во всех будущих режимах;
- что любая станция всегда пассивна;
- что вся платформа обязана физически быть набором отдельных микросервисов.

---

## 4. Рассмотренные альтернативы

### Альтернатива A: Ограничить базовый состав линейным сценарием только с шаттлом

Отклонена. Это не соответствует реальной поставке и не проверяет самые рискованные механизмы межуровневой оркестрации.

### Альтернатива B: Сразу детально описать семейство кранов-штабелёров

Отклонена. Сейчас это создаст объём абстракций, не дающих практической ценности для первой поставки.

### Альтернатива C: Вообще не фиксировать базовый состав, а описывать только возможности

Отклонена. Для старта реализации это слишком размыто и не даёт однозначных правил исполнения.
