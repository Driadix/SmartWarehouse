# ADR-011: Технологический базовый состав фазы 1

**Статус:** Принято  
**Дата:** 2026-04-04  
**Связанные артефакты:** ArchitecturalVision.md, Architecture-Baseline-Phase-1.md, ADR-004, ADR-008, ADR-009, ADR-010

---

## 1. Контекст

До настоящего момента в архитектуре были зафиксированы логические границы и контракты, но не был зафиксирован технологический базовый состав:

- какой именно серверный технологический стек считается базовым для `platform-core`;
- где проходит граница между `C#/.NET`, `Go` и `TypeScript`;
- нужны ли тяжёлые фреймворки сервисной шины и архитектура, построенная вокруг брокера, с первого дня;
- какие ранние технологические выборы считаются запрещёнными, даже если технически возможны.

Для фазы 1 это важно, потому что:

- команда уже неоднородна по стекам;
- экосистема `.NET` допускает слишком много разных направлений;
- ранняя неопределённость быстро превращается в несовместимые прототипы.

---

## 2. Решение

### 2.1. Язык и процесс исполнения для `platform-core`

Для `platform-core` принимается:

- `C#`
- `.NET 10 LTS`

Внутри `platform-core` базовыми компонентами процесса исполнения считаются:

- `ASP.NET Core` для `Northbound API`;
- `Microsoft.Extensions.Hosting` / `Generic Host` для фоновых обработчиков, проекций и задач доставки;
- `Microsoft.Extensions.DependencyInjection` как базовый DI-контейнер фазы 1;
- `OpenTelemetry .NET` для трассировок, метрик и лог-экспорта.

### 2.2. Доступ к данным в `platform-core`

Для работы с `PostgreSQL` принимаются:

- `Npgsql` как базовый драйвер;
- `EF Core` как базовый механизм миграций и основная ORM для модели записи и стандартных проекций состояния;
- прямой SQL через `Npgsql` допускается для доказанных критичных путей исполнения `WCS`, если это подтверждено профилированием и не ломает границы владения.

Для фазы 1 **не** принимаются как базовый состав:

- тяжёлые actor/framework-платформы (`Orleans`, `Akka.NET` и т.п.);
- фреймворки сервисной шины, скрывающие бизнес-семантику внутренних контрактов за транспортной абстракцией;
- обязательная зависимость от `Redis` как центрального хранилища состояния исполнения.

### 2.3. Контур интеграции нижнего уровня

Для адаптеров `ACL`, интеграции нижнего уровня и симулятора принимается:

- `Go`

Назначение этого контура:

- работа с сеансами устройств;
- завершение транспортов нижнего уровня;
- трансляция протоколов обмена;
- эмуляция симулятора и устройств;
- интеграции нижнего уровня и интеграции, зависящие от конкретной площадки, если они логически лежат на границе оборудования.

### 2.4. Базовый состав пользовательского интерфейса

Для цифрового двойника (`Digital Twin`) и операторского интерфейса в базовом составе фазы 1 принимаются:

- `TypeScript`
- `React`

Пользовательский интерфейс не является владельцем логики оркестрации и исполнения и не должен становиться вторым серверным ядром платформы.

### 2.5. База данных

Для фазы 1 принимается:

- `PostgreSQL`

База данных является обязательным инфраструктурным компонентом уже в первой версии. `SQLite` и хранение в памяти не считаются боевым базовым составом даже для первой поставки.

### 2.6. Межпроцессный транспорт

Для межпроцессного обмена между `platform-core`, контуром интеграции нижнего уровня и симуляционным контуром рекомендуемый базовый состав:

- `NATS JetStream`

Причины выбора:

- достаточно лёгкий для фазы 1;
- поддерживает публикацию-подписку, устойчивых подписчиков и модель запрос-ответ;
- хорошо подходит для обмена командами и событиями без перехода к тяжёлой архитектуре, изначально завязанной на Kafka;
- удобен для локальной разработки в контейнерах.

При этом:

- внутри `platform-core` `NATS` не заменяет внутренний прикладной контракт или шину;
- `NATS` является транспортной привязкой, а не источником бизнес-семантики.

### 2.7. Контейнеризация и локальная разработка

Для фазы 1 обязательны:

- `OCI/Docker`-совместимые контейнерные образы;
- локальный запуск полного окружения через сценарий наподобие Compose;
- единый способ поднятия `platform-core`, `PostgreSQL`, брокера, симулятора и стека наблюдаемости для разработчиков.

### 2.8. Тестовый базовый состав

Для `platform-core` базовый стек тестирования:

- `xUnit`;
- `Testcontainers` для интеграционных тестов с реальным `PostgreSQL` и брокером;
- контрактные тесты поверх `OpenAPI`, `AsyncAPI` и JSON Schema.

Для `Go`-контура:

- стандартный `go test`;
- интеграционные тесты с тем же брокером и теми же каноническими сообщениями.

### 2.9. Явно запрещённые ранние технологические выборы

Для фазы 1 не рекомендуются и не входят в базовый состав:

- `Kafka` как обязательный брокер с первого дня;
- полное событийное хранение как основная модель записи;
- обязательное развёртывание только на Kubernetes;
- отдельный микросервис на каждое устройство;
- серверная часть на `Node.js/TypeScript` как владелец жизненного цикла `Job` или `ExecutionTask`;
- прямой доступ на запись адаптеров оборудования в БД ядра.

## 3. Последствия

### 3.1. Что упрощается

- основное ядро, хранящее состояние, сосредоточено в одном серверном технологическом стеке;
- `Go` используется там, где он действительно даёт практическую пользу на границе оборудования;
- команда получает достаточно конкретный базовый состав по `.NET`, не уходя в ненужное разрастание стека фреймворков;
- транспорт и хранение становятся частью базового состава, а не случайным выбором первой реализации.

### 3.2. Что становится обязательным

- новые серверные модули должны либо укладываться в базовый состав, либо оформляться отдельным ADR;
- библиотечные исключения внутри `.NET` и `Go` должны быть явно мотивированы;
- межпроцессная интеграция должна оставаться контракт-ориентированной.

---

## 4. Рассмотренные альтернативы

### Альтернатива A: Вся серверная часть только на `C#/.NET`

Плюсы:

- минимальное число серверных технологических стеков;
- проще обмен людьми внутри серверной команды.

Минусы:

- интеграции нижнего уровня с протоколами обмена теряют естественную технологическую границу;
- слабее используется уже имеющаяся экспертиза команды в `Go`.

Не выбрана как базовая.

### Альтернатива B: `Go` для `WCS` и интеграции нижнего уровня, `C#` только для `WES`

Плюсы:

- естественная технологическая граница на стороне нижнего уровня;
- можно использовать сильные стороны `Go` на границе транспортного контура.

Минусы:

- усложняет самую критичную границу внутри хранящего состояние ядра платформы;
- требует более тяжёлого межъязыкового договора между `WES` и `WCS`;
- хуже использует текущую численную ёмкость команды `C#`.

Не выбрана как базовый состав фазы 1.

### Альтернатива C: `C#/.NET` для `platform-core`, `Go` для контура нижнего уровня и симулятора, `TypeScript/React` для пользовательского интерфейса

Плюсы:

- хорошо соответствует логическим границам системы;
- уменьшает число языков в самом критичном ядре, хранящем состояние;
- использует `Go` там, где он приносит максимальную пользу;
- хорошо совпадает с текущим составом команды и её масштабированием.

Минусы:

- остаются два серверных технологических стека вместо одного;
- нужны общие правила по наблюдаемости, контрактам и выпуску релизов между `C#` и `Go`.

Выбрана как рекомендуемая для фазы 1.
