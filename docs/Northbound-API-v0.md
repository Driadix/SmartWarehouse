# Northbound API v0
### `REST + static webhooks` для `Payload Transfer Job`

**Статус:** Живой документ  
**Последнее обновление:** 2026-04-04  
**Связанные артефакты:** ArchitecturalVision.md, Event-Catalog-v0.md, Topology-Configuration-Model-v0.md, docs/api/northbound/openapi-v0.yaml, docs/Standards/OpenAPI Specification v3.2.0.pdf

---

## 1. Назначение

Документ фиксирует минимальный внешний интеграционный контракт текущего базового состава.

Цели документа:

- дать `WMS` и другим внешним системам стабильный способ постановки и отслеживания задания на перенос грузовой единицы;
- не выводить наружу внутренние сущности `WCS`, семейственно-специфические runtime-фазы и операционные детали движения;
- зафиксировать минимальный `v0`, который можно развивать без размывания границы между внешним бизнес-контрактом и внутренним контуром исполнения.

---

## 2. Границы `v0`

`Northbound API v0` включает только:

- создание задания на перенос грузовой единицы между двумя внешними конечными точками;
- получение текущей проекции состояния задания;
- поиск задания по `clientOrderId`;
- запрос отмены задания;
- исходящую доставку webhook-событий `job.accepted` и `job.state_changed`.

`Northbound API v0` не включает:

- создание `charge`, `relocation` и иных типов `Job`;
- управление устройствами, станциями, лифтами и топологией;
- публикацию `ExecutionTask`, `RuntimePhase`, резервирований и `DeviceSession`;
- operator/admin API;
- сырую телеметрию и внутренние эксплуатационные события `WCS`;
- динамическую регистрацию webhook-адресов.

---

## 3. Паттерн интеграции

Паттерн взаимодействия фиксируется как:

`синхронный приём запроса -> асинхронный результат`

Правила `v0`:

- входящий поток реализуется через `REST`;
- исходящий поток реализуется через статически настроенные `webhook`;
- `GET`-операции служат для диагностики, повторного чтения и поддержки интеграции, но не являются базовым механизмом получения результата;
- URL webhook-приёмника, способ аутентификации и политика сетевого доступа задаются профилем развёртывания и не нормируются данным документом.

---

## 4. Внешняя ресурсная модель

Внешний ресурс `v0`:

```text
PayloadTransferJob {
  jobId
  clientOrderId
  state
  sourceEndpointId
  targetEndpointId
  priority
  payloadRef?
  attributes?
  reason?
  createdAt
  updatedAt
  completedAt?
}
```

Правила:

- ресурс описывает только бизнес-запрос на перенос грузовой единицы;
- `sourceEndpointId` и `targetEndpointId` являются внешними бизнес-адресами, а не `NodeId`;
- трансляция `endpointId` во внутренние узлы и станции выполняется через `Topology Configuration`;
- `payloadRef` допускается как опциональный внешний идентификатор груза;
- `attributes` являются расширяемым словарём и не трактуются платформой как обязательные инварианты `v0`.

---

## 5. Внешняя модель состояния

Во внешний контракт выводится только проекция верхнего уровня:

- `ACCEPTED` - запрос принят, задание создано и доступно для отслеживания;
- `IN_PROGRESS` - задание подтверждённо исполняется;
- `SUSPENDED` - задание временно остановлено и требует повторного согласования, ожидания или перепланирования;
- `COMPLETED` - задание завершено успешно;
- `FAILED` - задание завершилось неуспешно;
- `CANCELLED` - задание отменено.

Во внешний контракт `v0` сознательно не выводятся:

- `Created`;
- `Planned`;
- внутренние статусы `ExecutionTask`;
- `RuntimePhase`.

---

## 6. Идемпотентность и адресация

### 6.1. `clientOrderId`

- `clientOrderId` обязателен для каждого `POST /payload-transfer-jobs`;
- повторный запрос с тем же `clientOrderId` и тем же нормализованным телом обязан возвращать тот же `jobId`;
- повторный запрос с тем же `clientOrderId`, но другим нормализованным телом должен возвращать `409 Conflict`.

### 6.2. Нормализация тела для идемпотентности

Для `v0` нормализованным телом считаются следующие поля:

- `sourceEndpointId`;
- `targetEndpointId`;
- `priority`, где отсутствие эквивалентно `NORMAL`;
- `payloadRef`, где отсутствие эквивалентно пустому объекту;
- `attributes`, где отсутствие эквивалентно пустому объекту.

### 6.3. Адресация конечных точек

- `sourceEndpointId` и `targetEndpointId` обязаны ссылаться на `endpointMappings[]` из `Topology Configuration`;
- внешняя система не оперирует `StationId`, `NodeId` и `TransferPointId`;
- `sourceEndpointId` и `targetEndpointId` не должны совпадать.

---

## 7. Webhook-проекции

Во внешний `v0` публикуются только две проекции:

- `job.accepted` - внешняя проекция события `JobAccepted`;
- `job.state_changed` - внешняя проекция события `JobStateChanged`.

Правила доставки:

- доставка допускается по модели `at-least-once`;
- дедупликация на стороне получателя выполняется по `eventId`;
- причинная упорядоченность требуется только в пределах одного `jobId`;
- наружу публикуется стабильный внешний payload, а не сырая внутренняя форма канонического события.

Webhook-событие должно содержать:

- `eventId`;
- `eventType`;
- `eventVersion`;
- `occurredAt`;
- полную текущую проекцию `PayloadTransferJob`.

Для `job.state_changed` дополнительно передаётся `previousState`.

## 8. Каталог операций

### 8.1. `POST /api/v0/payload-transfer-jobs`

Назначение:

- создать новое `PayloadTransferJob`.

Успешный исход:

- `202 Accepted`, если запрос принят и новое задание создано;
- `200 OK`, если это идемпотентный повтор уже принятого эквивалентного запроса.

### 8.2. `GET /api/v0/payload-transfer-jobs/{jobId}`

Назначение:

- получить текущую внешнюю проекцию задания по `jobId`.

### 8.3. `GET /api/v0/payload-transfer-jobs/by-client-order/{clientOrderId}`

Назначение:

- найти задание по клиентскому идентификатору.

### 8.4. `POST /api/v0/payload-transfer-jobs/{jobId}/cancel`

Назначение:

- запросить отмену задания.

Успешный исход:

- `202 Accepted`, если запрос отмены принят;
- `200 OK`, если задание уже находится в состоянии `CANCELLED`.

`409 Conflict` возвращается, если задание уже находится в другом терминальном состоянии и отмена больше невозможна.

---

## 9. Что сознательно отложено

- API регистрации webhook-подписок;
- batch-операции;
- отдельный operator/admin API;
- единый обязательный профиль аутентификации и авторизации;
- создание других типов `Job` поверх того же внешнего контракта;
- потоковая подписка на события вместо webhook;
- экспорт внутренних событий `WCS`.
