# Глоссарий терминов
### Умный склад — платформа WES/WCS

**Статус:** Нормативный словарь терминов  
**Последнее обновление:** 2026-04-03  
**Связанные артефакты:** ArchitecturalVision.md, Architecture-Baseline-Phase-1.md, ADR-001, ADR-002, ADR-003, ADR-004, ADR-005, ADR-006, DomainModel-v0

## 1. Назначение

Настоящий глоссарий фиксирует предпочтительные термины, допустимые английские эквиваленты и правила их употребления в документации проекта.

Цели глоссария:

- обеспечить единый профессиональный русский язык в документации;
- сократить количество необоснованных англицизмов;
- сохранить английские обозначения там, где они являются частью кода, общепринятого технического обозначения или имени системной сущности;
- исключить ситуацию, когда один и тот же термин в разных документах описывается разными словами.

---

## 2. Правила употребления

### 2.1. Общие правила

- По умолчанию в тексте используется **русский термин**.
- Английский термин допускается, если он:
  - является именем системной сущности, типа, поля, события или файла;
  - не имеет краткого и устойчивого технического русского аналога;
  - нужен для точного соответствия коду, API, схеме сообщений или отраслевому стандарту.
- При первом употреблении предпочтительно использовать форму: **русский термин (английский термин)**.
- В дальнейшем в повествовательном тексте следует использовать русский термин, если контекст однозначен.
- Идентификаторы кода, событий, полей и перечислений сохраняются в исходном виде и выделяются обратными кавычками: `Job`, `ExecutionTask`, `NodeReached`, `currentNode`.

### 2.2. Чего следует избегать

Следует избегать:

- англицизмов, используемых без необходимости в обычном описательном тексте;
- смешения русского и английского в одном и том же обороте без пояснения;
- разговорных и расплывчатых слов вроде «магический», «просто едет», «как бы» и им подобных в нормативной документации;
- подмены архитектурных терминов названиями конкретной реализации, если это не зафиксировано как текущий базовый состав.

### 2.3. Правила для кодовых имён

Названия классов, типов устройств, режимов, событий и полей допускается оставлять на английском языке, если они используются как канонические идентификаторы системы. В таких случаях в тексте рядом должно быть русское объяснение назначения сущности.

Пример:

- `Shuttle3D` — трёхмерный шаттл, способный двигаться по ярусу с ветвлениями и поворотами.
- `Node` — узел графа, дискретная точка логического позиционирования.

## 3. Термины предметной области

| Предпочтительный термин | Допустимый английский эквивалент | Определение и правило употребления |
|---|---|---|
| Узел | `Node` | Дискретная точка логического позиционирования в графе склада. В коде и схемах допускается `Node`; в тексте предпочтительно «узел». |
| Ребро графа | `Edge` | Направленная связь между узлами графа. |
| Узел ветвления | `SwitchNode` | Узел, в котором возможен выбор направления движения и требуется исключительное краткосрочное резервирование. |
| Точка передачи | `TransferPoint` | Точка согласованной передачи ресурса или груза между участниками операции. |
| Узел вертикального перевозчика | `CarrierNode` | Узел графа, соответствующий положению вертикального перевозчика на конкретном уровне. |
| Узел станции | `StationNode` | Узел графа, соответствующий точке примыкания станции загрузки или выгрузки. |
| Подвижная транспортная единица | `MobileTransportUnit` | Базовое семейство подвижных ресурсов, перемещающих груз по графу склада. |
| Вертикальный перевозчик | `VerticalCarrier` | Базовое семейство ресурсов вертикального перемещения между уровнями. |
| 3D-шаттл | `Shuttle3D` | Конкретный тип подвижной транспортной единицы в текущем базовом составе реализации. Английское имя допустимо как имя системной сущности. |
| Гибридный лифт | `HybridLift` | Конкретная реализация вертикального перевозчика, способная перевозить шаттл и, при наличии соответствующего режима, груз. |
| Граница станции | `StationBoundary` | Доменная граница взаимодействия платформы с зоной загрузки, выгрузки или иной внешней точкой передачи. |
| Пассивная граница станции | `Passive StationBoundary` | Граница станции без собственного нижнего сеанса управления. В текущем базовом составе `LoadStation` и `UnloadStation` работают по этой модели. |
| Станция загрузки | `LoadStation` | Точка входа груза в систему. |
| Станция выгрузки | `UnloadStation` | Точка выхода груза из системы. |
| Грузовая единица | `Payload` | Физическая единица груза, для которой строится сквозной маршрут и отслеживается физическое удержание. |
| Задание верхнего уровня | `Job` | Макронамерение системы, владеемое `WES`. В повествовательном тексте допустимо также «задание» при однозначном контексте. |
| Исполнительная задача | `ExecutionTask` | Атомарный шаг исполнения, назначенный конкретному ресурсу или границе станции. |
| Возможность | `Capability` | Аппаратная или активная функциональная возможность ресурса. |
| Отказ | `Fault` | Формализованное сообщение о неисправности, деградации или запрещающем состоянии ресурса. |
| Сеанс устройства | `DeviceSession` | Управляемое соединение устройства с контуром исполнения. |
| Снимок состояния | `Snapshot`, `StateSnapshot` | Полное или согласованное состояние объекта на определённый момент времени. |
| Проекция состояния | `Read-model` | Материализованное представление состояния для чтения, визуализации и анализа. |
| Цифровой двойник | `Digital Twin` | Подсистема визуализации и аналитической проекции состояния склада. |

## 4. Термины управления и исполнения

| Предпочтительный термин | Допустимый английский эквивалент | Определение и правило употребления |
|---|---|---|
| Оркестрация | `Orchestration` | Координация исполнения нескольких ресурсов и шагов на уровне системы. Русский термин предпочтителен. |
| Исполнение | `Execution` | Фактическое выполнение шагов задания в контуре `WCS`. |
| Скользящее окно резервирования | `Sliding Window` | Механизм выдачи устройству ограниченного окна разрешённого движения по ближайшим узлам. |
| Краткосрочное резервирование | `Short-horizon Reservation` | Резервирование узлов и участков маршрута на ближайший исполнимый горизонт. |
| Операция передачи | `Handoff` | Координированная передача ресурса или груза между участниками процесса. |
| Режим передачи | `Transfer Mode` | Формализованный сценарий передачи между ресурсами и/или станциями. |
| Физическое удержание груза | `Payload Custody` | Факт того, какой ресурс или станция физически удерживает груз в текущий момент. |
| Контроль доступности | `Liveness` | Контроль того, что ресурс остаётся управляемым и доступным в контуре исполнения. |
| Карантинная изоляция | `Quarantine` | Исключение проблемного участка графа или ресурса из нормального контура исполнения до восстановления. |
| Повторная попытка | `Retry` | Повторное выполнение шага или операции по правилам системы. |
| Компенсирующее действие | `Compensation` | Действие, откатывающее или безопасно завершающее ранее начатую операцию. |
| Политика владения | `Ownership Model` | Формализованное распределение ответственности и полномочий между `WES`, `WCS`, `ACL` и устройствами. |
| Правила исполнения | `Execution Semantics` | Набор правил, определяющих, как семейство ресурсов исполняет задачи и подтверждает результат. |
| Контроллер семейства ресурсов | `Family Controller` | Модуль контура `WCS`, реализующий правила исполнения для конкретного семейства ресурсов. |
| Материализация исполнения | `Execution Materialization` | Преобразование планового `ExecutionTask` во внутренние runtime-фазы и канонические команды внутри `WCS`. |
| Повторное согласование состояния | `Reconciliation` | Согласование фактического состояния ресурса и шага исполнения по `StateSnapshot` после потери сеанса, рестарта или деградации связи. |

## 5. Термины интеграции и архитектуры

| Предпочтительный термин | Допустимый английский эквивалент | Определение и правило употребления |
|---|---|---|
| Верхний интеграционный интерфейс | `Northbound API` | Интерфейс взаимодействия платформы с WMS и другими внешними системами верхнего уровня. |
| Нижний интеграционный интерфейс | `Southbound API` | Интерфейс взаимодействия `WCS` с устройствами и адаптерами оборудования. |
| Канонический нижний контракт интеграции | `Southbound Contract` | Единый смысловой контракт команд, состояний и событий нижнего уровня. |
| Адаптерный слой изоляции | `ACL`, `Anti-Corruption Layer` | Слой преобразования канонических команд и событий платформы во внешние протоколы и обратно. |
| Транспортно-независимый | `Transport-agnostic` | Не зависящий от конкретного сетевого или проводного транспорта передачи сообщений. |
| Ориентированный на логические узлы | `Node-based` | Основанный на работе с `NodeId`, а не с сырыми координатами. |
| Основанный на сеансах управления | `Session-based` | Использующий модель управляемого сеанса устройства с контролем продолжительности и доступности. |
| Идемпотентный | `Idempotent` | Повторный вызов с теми же идентификаторами не изменяет итоговый результат. |
| Сигнал доступности | `Heartbeat` | Регулярное сообщение, подтверждающее, что устройство и его сеанс остаются активными. |
| Сеанс с ограниченным сроком действия | `Lease Semantics` | Модель, при которой сеанс считается действительным только при своевременном подтверждении его продолжения. |
| Базовый состав реализации | `Baseline` | Формально зафиксированный текущий состав поддерживаемых сущностей, режимов и ограничений реализации. |
| Активированный режим | `Enabled Mode` | Режим, включённый в текущем базовом составе реализации. |
| Аппаратная возможность | `Hardware Capability` | Возможность, поддерживаемая устройством физически или на уровне локального контроллера, независимо от того, включена ли она в текущую версию платформы. |
| Имитационное моделирование в изолированном контуре | `Sandbox Simulation` | Среда воспроизводимого моделирования поведения системы без влияния на боевой контур. |

## 7. Порядок обновления

Если в проект добавляется новая системная сущность, новый тип ресурса, новое событие или новый режим передачи, сначала следует:

1. определить предпочтительный русский термин;
2. определить, нужен ли английский эквивалент как системное имя;
3. добавить термин в настоящий глоссарий;
4. только после этого использовать его в ADR, архитектурных документах и спецификациях.
