# Интеграция станций и площадки v0
### Пассивные `LoadStation` / `UnloadStation` в базовом составе `Phase 1`

**Статус:** Живой документ  
**Последнее обновление:** 2026-04-04  
**Связанные артефакты:** ArchitecturalVision.md, Architecture-Baseline-Phase-1.md, Execution-Semantics-v0.md, ADR-003, ADR-004

---

## 1. Назначение

Документ фиксирует минимальную роль `Station/Site Integration Adapter` для пассивных станций.

Цель документа:

- не превращать пассивную станцию в активное устройство с собственным `DeviceSession`;
- не размывать границу между `WCS`, `ACL` и site-specific интеграцией;
- дать `WCS` явный источник готовности станции и подтверждённых фактов передачи на её границе.

---

## 2. Область действия

В текущем базовом составе документ покрывает только:

- пассивную `LoadStation`;
- пассивную `UnloadStation`.

Документ не покрывает:

- `ChargeNode` и `ServiceNode`, для которых `NodeReached` уже достаточно в текущем базовом составе;
- активные станции с собственным нижним сеансом управления;
- внутренний wire-протокол конкретного PLC, HMI, датчика или внешнего site-сервиса.

---

## 3. Роль адаптера

`Station/Site Integration Adapter` является интеграционным модулем, который:

- получает site-specific сигналы о готовности пассивной станции;
- получает подтверждённые факты передачи на границе станции;
- нормализует эти сигналы до канонических фактов, понятных `WCS`;
- не открывает `DeviceSession` станции;
- не исполняет команды движения и не подменяет `ACL`.

Адаптер может быть развёрнут:

- как модуль внутри `WCS`;
- как отдельный процесс;
- как внешний интеграционный сервис.

Физическая схема не меняет логическую роль адаптера.

---

## 4. Минимальные канонические факты

### 4.1. `StationReadinessChanged`

```text
StationReadinessChanged {
  stationId
  readiness: READY | BLOCKED | OFFLINE | MAINTENANCE
  reason?
  observedAt
}
```

Смысл:

- отражает, может ли `WCS` начинать или продолжать `StationTransfer`;
- не завершает передачу сам по себе.

### 4.2. `StationTransferFactReported`

```text
StationTransferFactReported {
  stationId
  direction: LOAD_TO_SHUTTLE | SHUTTLE_TO_UNLOAD
  factType: CUSTODY_TRANSFER_CONFIRMED
  payloadId?
  correlationId?
  observedAt
}
```

Смысл:

- подтверждает факт физической передачи на границе пассивной станции;
- используется `WCS` для фиксации `PayloadCustodyChanged`;
- не создаёт и не завершает `ExecutionTask` напрямую.

---

## 5. Правила интерпретации

1. `NodeReached(StationNode)` подтверждает только позиционирование шаттла.
2. `StationReadinessChanged(READY)` подтверждает, что граница станции готова к передаче.
3. `StationTransferFactReported(CUSTODY_TRANSFER_CONFIRMED)` подтверждает физический факт передачи.
4. Только после пункта 3 `WCS` вправе публиковать `PayloadCustodyChanged` и завершать `StationTransfer`.
5. Если adapter-fact противоречит текущему направлению шага, `PayloadCustody` или положению шаттла, `WCS` обязан эскалировать на повторное согласование или операторское решение.

---

## 6. Что сознательно не фиксируется сейчас

- внутренний транспорт между адаптером и внешним источником сигнала;
- полный каталог причин `reason`;
- сложные сценарии буферов станции больше единицы;
- многократные промежуточные факты передачи;
- операторские UI-потоки подтверждения.
