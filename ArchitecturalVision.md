## 1. Назначение документа

### 1.1. Цель и область применения

Настоящий документ архитектурного видения (Architecture Vision) фиксирует архитектурный фундамент, глобальные границы ответственности, предметную область (доменную модель) и базовые интеграционные контракты разрабатываемого программного комплекса управления автоматизированным складом «Умный склад» (далее — Система, класс WES/WCS).

### 1.2. Целевая аудитория

Основная аудитория документа — системные архитекторы, ведущие разработчики программного обеспечения, специалисты по интеграции с внешними системами управления складом (WMS), разработчики аппаратных адаптеров, руководители проекта, инженеры-конструкторы и технологи. Для них документ служит единым источником информации о том, где заканчивается ответственность программного ядра Системы и начинаются физические процессы и управление оборудованием.

### 1.3. Уровень детализации и связанные артефакты

Документ задаёт концептуальный и логический уровни архитектуры. Детали сериализации, структуры данных, форматы полей и физические схемы баз данных определяются в профильных спецификациях:

- **Спецификация предметной области (Domain Model)**
- **Конфигурация топологии и маршрутизации (Topology Config)**
- **Каталог событий (Event Catalog)**
- **Спецификация интеграционных интерфейсов (API Spec)**
- **ADR-001:** Модель лифта, режимы шаттла и трассируемость при межъярусных операциях

## 2. Границы системы и контекст

Система «Умный склад» представляет собой гибридный программный комплекс (WES/WCS), предназначенный для оркестрации парка складской робототехники (шаттлы, лифты, краны-штабелёры). Архитектура базируется на строгом разделении логического управления и физического исполнения.

### 2.1. Зона ответственности (In Scope)

Программное ядро Системы (WES) обеспечивает реализацию следующих функций:

- **Глобальная маршрутизация (Topological Routing + Dynamic Spatial Locking):** Динамическое вычисление маршрутов на едином логическом графе с пространственным резервированием узлов по принципу «скользящего окна».

- **Диспетчеризация и оркестрация:** Декомпозиция бизнес-задач от WMS на макро- и микрозадачи (Saga Pattern) и их оптимальное распределение между доступными ресурсами.

- **Управление межъярусными операциями:** Оркестрация перемещения шаттлов и паллет между ярусами через лифты, включая координацию State Machine Handshake при посадке/высадке. Подробности — ADR-001.

- **Аппаратная абстракция:** Унификация взаимодействия с парком оборудования различных типов через стандартизированный слой антикоррупционных адаптеров (Anti-Corruption Layer).

- **Управление исключениями топологии:** Изоляция проблемных участков графа (карантинные зоны) на основе телеметрии и срабатывания Heartbeat-таймаутов от оборудования, а также локальный перерасчёт маршрутов.

### 2.2. Строгие исключения (Out of Scope)

Из контура Системы строго исключены следующие функции:

- **Управление товародвижением (WMS-контур):** Система не ведёт учёт складских запасов, не управляет клиентскими заказами и партиями.

- **Функциональная безопасность (Functional Safety):** Система не является компонентом обеспечения безопасности. Предотвращение физических столкновений, экстренные остановки и реакция на препятствия реализуются исключительно на аппаратном уровне бортовыми датчиками устройств и локальными контроллерами (Hardware Interlocks).

- **Низкоуровневое управление оборудованием:** Система не формирует кинематические профили движения и не опрашивает аппаратные сенсоры. Взаимодействие строится на асинхронном обмене высокоуровневыми командами с адаптерами.

- **Физическое позиционирование:** Определение текущих пространственных координат делегировано исполнительным устройствам. Программное ядро оперирует только логическими узлами графа (Node ID). Трансляция координат (QR, RFID, Lidar) в логические узлы происходит в слое адаптеров.

- **Управление энергопотреблением в реальном времени:** Ядро Системы не осуществляет микроменеджмент батарей. Деградация энергопитания транслируется в логическое отключение возможностей ресурса (Dynamic Capabilities).

## 3. Архитектурный обзор подсистем

Архитектура строится на принципах микросервисной декомпозиции и событийно-ориентированного взаимодействия (Event-Driven Architecture).

### 3.1. Информационные шины (Data Buses)

Для предотвращения деградации производительности потоки данных физически разделены:

- **Command & Event Bus (Транзакционная шина):** Брокер сообщений с гарантированной доставкой для критических бизнес-событий (TaskCreated, StatusChanged, HardwareAlarm). Порядок событий гарантируется логическими часами (Sequence Numbers).

- **Telemetry Firehose (Шина телеметрии):** Высокоскоростной брокер для трансляции «сырых» данных в реальном времени (потоковые координаты, статус батарей). Используется метка времени устройства (синхронизация по локальному NTP). Применяется для визуализации и предикативной аналитики. **Метки времени используются исключительно для аналитики и визуализации — никогда для управляющей логики.**

### 3.2. WES (Warehouse Execution System) — Ядро оркестрации

Единый логический центр принятия решений. Реализован как **Stateful In-Memory потоковый процессор с применением Event Sourcing (IMDG)**. Состояние топологии и блокировок удерживается в оперативной памяти для обеспечения микросекундных задержек при поиске путей (алгоритмы A\*/Dijkstra). Каждое изменение графа порождает доменное событие (State Delta). Восстановление после сбоев (RTO < 5 мин) осуществляется путём загрузки последнего персистентного снапшота (Base Snapshot) из БД и «накатывания» последних событий из брокера.

### 3.3. WCS Fleet Managers (Менеджеры флота) и ACL

Слои управления флотом (WCS) и аппаратной абстракции (ACL) объединены в специализированные микросервисы под каждый тип оборудования.

- **Трансляция и Маппинг (ACL):** Преобразует универсальные атомарные команды Системы во внутренние протоколы устройств.

- **Диспетчеризация (WCS):** Управляет буферизацией узлов и динамическими блокировками пространства (Sliding Window), а также осуществляет мониторинг доступности (Heartbeat/LWT), защищая ядро WES от сетевых аномалий железа. WCS лифта дополнительно оркестрирует State Machine Handshake при межъярусных операциях и несёт ответственность за переключение `movementMode` шаттла.

### 3.4. Digital Twin (Цифровой Двойник)

Система 3D-визуализации и аналитики, не имеющая прямого влияния на контур управления.

- **Live Monitoring:** При старте запрашивает у WES актуальный базовый слепок состояния (Base Snapshot) через REST/gRPC API, после чего подписывается на транзакционную шину и шину телеметрии, применяя дельта-изменения (State Deltas) для построения локальной проекции склада в реальном времени. При получении события `LiftMoved` Digital Twin обновляет `currentNode` для лифта **и** для всех шаттлов, находящихся внутри него в режиме `PASSENGER`, — полная трассируемость всего парка обеспечивается всегда.

- **Sandbox Simulation (Режим симуляции):** Изолированная среда, работающая поверх теневого ядра (Shadow WES). Позволяет валидировать алгоритмы маршрутизации через программные моки (заглушки) аппаратных адаптеров до появления реального оборудования.

### 3.5. Архитектура управления движением

Во избежание коллизий Система работает с топологической маршрутизацией с динамическим резервированием пространства.

- **Уровень планирования (WES):** Прокладывает глобальный маршрут, оценивая общую загруженность магистралей. При маршрутах через несколько ярусов WES прокладывает сквозной путь через `LiftNode` — WCS транслирует его в Step-последовательность с операциями `BoardLift` / `ExitLift`.

- **Уровень исполнения (WCS):** Ведёт карту занятости. Выдаёт устройству разрешение на движение только на выделенный пул ближайших свободных узлов (Sliding Window).

- **Аппаратный уровень (Устройство):** Контроллер берёт на себя физическое исполнение и аппаратный Fail-Safe: **при неполучении Heartbeat-пакетов (Ping/Ack) от WCS устройство обязано выполнить экстренную остановку** для предотвращения неуправляемого движения в слепых зонах связи.

## 4. Доменная модель: Задачи и Ресурсы

Доменная модель ядра концептуально изолирована от бизнес-процессов инвентаризации (WMS) и аппаратных характеристик (ACL).

### 4.1. Концепция Задач (Task & Job Model)

Во избежание «ослепления» ядра, комплексные процессы делятся на уровни согласно **паттерну Saga (Оркестрация)**:

1. **Макро-задача (Job):** Комплексное намерение (Intent). WES жёстко резервирует логический ресурс под весь Job. Примеры:
   - *Payload Job:* Перемещение к ячейке A → Взять паллету → Отвезти в зону B (возможно через несколько ярусов) → Положить паллету.
   - *Charge Job:* Перемещение к станции зарядки (возможно на другом ярусе через лифт) → Стыковка → Зарядка → Отстыковка.
   - *Relocation Job:* Перемещение пустого шаттла в зону ожидания для балансировки парка.

2. **Атомарные логические Задачи (Atomic Tasks):** Job декомпозируется в строгую последовательность микрокоманд:
   - **Transport Task (Move):** Позиционирование ресурса между узлами графа (чистая навигация).
   - **Action Task:** Атомарная аппаратная операция без смены узла — захват/сброс груза, `BoardLift`, `ExitLift`, стыковка с зарядной станцией, активация конвейера.

Каждая атомарная задача и Job строго следуют циклу состояний: **Создана → Спланирована → На исполнении → Приостановлена (Suspended) → Завершена / Ошибка / Отменена (Cancelled) → Компенсируется (Compensating) → Компенсирована / Сбой компенсации**.

Наличие статусов приостановки, отмены и компенсации позволяет WES безопасно прерывать Job (например, при отзыве заказа WMS или перехвате шаттла на более приоритетную задачу), дождавшись завершения текущего атомарного шага.

### 4.2. Концепция Ресурсов и Динамических Возможностей (Capabilities)

Ядро распределяет задачи исключительно путём сопоставления требований задачи и профиля Ресурса (Set Intersection).

Все управляемые устройства (шаттлы, лифты, краны-штабелёры) являются наследниками базовой сущности `Device`:

#### Шаттл (Shuttle extends Device)

Шаттл обладает двухосевой моделью состояния (подробности — ADR-001):

```
Shuttle {
  movementMode:   AUTONOMOUS | PASSENGER
  dispatchStatus: AVAILABLE | OCCUPIED | SUSPENDED | MAINTENANCE
  carrierId:      <liftId> | null   // активно при movementMode = PASSENGER
}
```

- **`movementMode`** описывает *кто управляет физическим движением*: в режиме `PASSENGER` шаттл не получает команд движения, его `currentNode` наследуется от лифта-носителя в реальном времени.
- **`dispatchStatus`** описывает *доступность для диспетчеризации WES*: шаттл может быть `PASSENGER` и при этом `OCCUPIED` (Job продолжается) или `MAINTENANCE` (едет на сервис).

Job шаттла **не прерывается и не передаётся** при въезде в лифт — меняется только `movementMode`. Шаттл остаётся владельцем Job и паллеты на всём протяжении межъярусного перемещения.

#### Лифт (Lift extends Device)

Лифт моделируется **исключительно как `Device`** — не как `Edge` и не как гибридная сущность (подробности и обоснование — ADR-001):

```
Lift {
  liftClass:          PALLET_ONLY | SHUTTLE_ONLY | HYBRID
  currentFloor:       Floor_N_LiftNode
  passengerShuttleId: <shuttleId> | null
}
```

- `PALLET_ONLY` — перевозит только паллеты.
- `SHUTTLE_ONLY` — перевозит шаттлы (с грузом или без).
- `HYBRID` — перевозит и паллеты, и шаттлы, один слот кабины.

Один склад может содержать лифты разных классов одновременно. Ядро выбирает лифт по `Capabilities` под требования конкретного Job.

#### Динамическое маскирование (Dynamic Masking)

Для обработки физической деградации (низкий заряд, частичная поломка) внедрена модель **Dynamic Masking**:

- **Hardware Capabilities:** Статичные аппаратные возможности (габариты, грузоподъёмность, `liftClass`).
- **Active Capabilities:** Текущие доступные возможности. При деградации WCS публикует событие `CapabilitiesDegraded`, отключая соответствующий признак. Ядро WES перестаёт выдавать ресурсу несовместимые задачи без усложнения алгоритмов маршрутизации.

### 4.3. Синхронизация транзитных операций и передачи груза (Handoff)

Физическая синхронизация при межъярусных операциях и передаче груза между разнородными устройствами не опирается на системное время. WCS реализует механизм **«Координированного рукопожатия» на базе конечных автоматов (State Machine Handshake)**:

1. Агент 1 (Шаттл) сигнализирует `ReadyToBoard` / `ReadyToUnload`.
2. Агент 2 (Лифт) сигнализирует `ReadyToReceive`.
3. Только получив оба подтверждения (строгая причинно-следственная связь, независимая от сети), WCS выдаёт команду на старт приводов.
4. В случае сбоя одного из агентов — WCS безопасно отменяет транзакцию для второго, оба переходят в `SUSPENDED`.

## 5. Модель топологии и конфигурации

Физическое пространство склада концептуализируется ядром исключительно как математический граф.

### 5.1. Иерархический связный граф

- **Плоские графы ярусов (2D-подграфы):** Детерминированная сеть проездов на каждом ярусе.

- **Лифтовые узлы (Lift Nodes):** Лифтовая шахта представлена набором узлов — по одному на каждый ярус (`Floor_N_LiftNode`). Рёбра между `Floor_N_LiftNode` присутствуют в графе статически, помечены `traversalMode: LIFT_ONLY` и доступны только через `Lift Device`. A\*/Dijkstra учитывает эти рёбра при построении межъярусных маршрутов; WCS блокирует прямое использование этих рёбер шаттлами.

- **Transfer Node:** Специальный узел примыкания к лифту, обязательный на каждом ярусе у каждой лифтовой шахты. Единственная точка входа/выхода из лифта для шаттлов и паллет. Именно здесь происходит State Machine Handshake и смена `movementMode`.

### 5.2. Грузоориентированная маршрутизация

Глобальная маршрутизация (WES) оперирует сквозным маршрутом для **целевого груза**. Трансляция абстрактного маршрута груза в микрозадачи для разных шаттлов и лифта делегируется Менеджеру флота (WCS).

### 5.3. Узлы и Рёбра

- **Узел (Node):** Дискретная точка позиционирования. Вместимость — одна логическая единица флота. Типы узлов: стандартный, `TransferNode`, `Floor_N_LiftNode`, `ServiceNode`, `ChargeNode`.

- **Ребро (Edge):** Направленный вектор перемещения между узлами со взвешенными коэффициентами и атрибутом `traversalMode` (OPEN | LIFT_ONLY | RESTRICTED).

### 5.4. Трансляция бизнес-адресов

Ядро изолировано от адресации WMS (Ряд-Ярус-Место). **Topology Compiler** маппит складские локации на узлы (NodeId) на основе конфигурационных макро-правил при развёртывании. Трансляция бизнес-адресов происходит на пограничном API Gateway.

## 6. Событийная модель (Event-Driven Architecture)

### 6.1. Управление состоянием (Outbox Pattern)

Для обеспечения гарантии доставки применяется паттерн **Transactional Outbox**: изменение In-Memory состояния и формирование исходящего события происходят атомарно, после чего фоновый релей публикует событие в транзакционную шину.

### 6.2. Гарантии доставки, Идемпотентность и Порядок событий

- **Sequence Numbers (Логические часы):** Критические бизнес-события нумеруются инкрементальными счётчиками для гарантии строгого порядка обработки.
- **At-Least-Once Delivery:** Брокер гарантирует доставку как минимум один раз.
- **Идемпотентность:** Дублированные сигналы безопасно игнорируются на основе Correlation ID.

### 6.3. Канонические события (Event Envelope)

События состоят из конверта с метаданными (UUID, Sequence ID, Correlation ID для сквозной трассировки от WMS до оборудования) и полезной нагрузки (Payload). Ключевые события межъярусных операций:

```
ShuttleMovementModeChanged  { shuttleId, from, to, carrierId?, timestamp }
ShuttleBoarded              { shuttleId, liftId, floor, jobId }
ShuttleExited               { shuttleId, liftId, floor, jobId }
LiftCapabilityDeclared      { liftId, liftClass, supportedFloors }
CapabilitiesDegraded        { deviceId, removedCapabilities, reason }
```

Полный каталог событий — в отдельном артефакте Event Catalog.

## 7. Интеграционные контракты

### 7.1. Northbound API (WMS ↔ Система)

Паттерн: «Синхронный приём — Асинхронный ответ».

- **Входящий поток:** WMS создаёт ордера через REST / gRPC с обязательным уникальным ключом `clientOrderId` (защита от дублей).
- **Исходящий поток:** Полное отсутствие поллинга. Статусы возвращаются в WMS асинхронно через Webhooks / Event Streaming.

### 7.2. Southbound API (WCS ↔ Устройства / ACL)

- **Механизм контроля доступности (Dead Zones):** Использование **MQTT LWT (Last Will and Testament)**. При исчезновении устройства из сети брокер автоматически генерирует событие потери связи. WCS немедленно переводит выделенные устройству узлы в статус «Карантинной зоны».

- **Протоколы:**
  - Для высоконагруженной рельсовой кинематики (шаттлы) — легковесный MQTT/UDP протокол, оптимизированный под Sliding Window Authorization с жёстким двусторонним Heartbeat контролем.
  - Протокол для собственных устройств — TBD (отдельный артефакт Device Protocol Spec).

---

## 8. Нефункциональные требования (NFR)

- Алгоритмическое ядро маршрутизации (In-Memory) должно обеспечивать диспетчеризацию до **1 000 одновременно активных логических задач**.
- Компонент WCS обязан обеспечивать опережающую буферизацию маршрутов (Chunking).
- **Целевая доступность (SLA):** 98%.
- **Recovery Time Objective (RTO):** Не более 5 минут (холодный старт базы, подгрузка Snapshot и Event Deltas).
- Телеметрия аппаратуры и логического состояния экспортируется в формате Prometheus/OpenTelemetry.
